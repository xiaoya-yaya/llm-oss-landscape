## Why

The weekly update pipeline currently produces separate internal and reader-facing reports (`weekly_report.md` and `yuque_report.md`), which causes duplicate content, inconsistent dates/counts, and manual handoff between report generation and publishing. The `--check` flow should instead generate one complete, deep weekly report and publish that same source of truth to Yuque, DingTalk, and GitHub PR in channel-appropriate forms.

## What Changes

- Generate one canonical weekly report from `--check` that combines deep trend insights, highlighted projects, complete review candidate tables, and per-project evidence.
- Call an LLM from inside the script to produce the deep trend insight sections automatically from structured project context.
- Archive every generated report under `reports/weekly_reports_by_agents/YYYY-MM-DD-weekly-agentic-ai-report.md` instead of only overwriting `data/weekly_report.md`.
- Keep `data/weekly_report.md` as the latest-copy compatibility output while treating the dated report under `reports/weekly_reports_by_agents/` as the canonical archive.
- **BREAKING**: Stop generating `data/yuque_report.md`; Yuque publishes the canonical weekly report content directly.
- Publish the full canonical report to Yuque before sending DingTalk.
- Create a GitHub PR after Yuque publish. The PR body contains an independent review checklist plus the full canonical weekly report content. The PR must not rely on Yuque links because Yuque documents may be inaccessible outside the internal network.
- Send DingTalk last, after both Yuque and PR are available. DingTalk stays concise, includes at most 3 highlighted projects, and emphasizes Yuque URL and PR URL.
- Defer deeper redesign of `--post-merge`; this change only makes the checklist block stable and independent so later parsing can improve safely.

## Capabilities

### New Capabilities

- `canonical-weekly-report`: A single complete weekly report artifact generated by `--check`, archived by date, and suitable for both reader-facing publishing and PR review.
- `llm-trend-generation`: Script-native LLM generation of deep trend insights from structured weekly project context with deterministic fallback behavior.
- `multi-channel-weekly-publishing`: Channel-specific publishing from the canonical report to Yuque, GitHub PR, and DingTalk in a fixed order.

### Modified Capabilities

## Impact

- `scripts/weekly_update.py`: Major update to report generation, LLM integration, report file paths, Yuque publishing, DingTalk notification order/content, and PR body construction.
- `scripts/test_weekly_update.py`: Add tests for report path generation, report structure, DingTalk summary limits, PR checklist/full-report composition, and LLM fallback behavior.
- `data/weekly_report.md`: Remains as latest compatibility copy.
- `data/yuque_report.md`: No longer generated by `--check`; may be removed or left stale until cleanup.
- `reports/weekly_reports_by_agents/`: New report archive directory.
- Environment: Requires LLM configuration such as `OPENAI_API_KEY` and optional `OPENAI_MODEL`; existing GitHub, ClickHouse, Yuque MCP/DWS, and DingTalk webhook credentials remain relevant.
